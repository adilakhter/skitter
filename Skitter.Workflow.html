    Skitter v0.1.0-dev Skitter.Workflow View Source

    Tools to interact with skitter workflows.

    This module defines an API that can be used to fetch data from skitter workflow structures. Besides this, this module defines the workflow/1 macro which can be used to define a skitter workflow.

    destination() :: {workflow_identifier(), Skitter.Component.port_name()}

    Address a token can be sent to.

    Specified as the combination of a valid workflow identifier that refers to a component instance and the name of a valid port of that component.

    Link to this type outgoing_links() View Source 
    outgoing_links() :: [{Skitter.Component.port_name(), [destination()]}]

    Outgoing links of a component

    Specified as a keyword list. Each key in the list corresponds to an out port of the component, while the values for this key correspond to all the t:destination ports this out port is connected with.

    Link to this type proto_instance() View Source 
    proto_instance() :: {Skitter.Component.t(), any(), outgoing_links()}

    Proto-instance type.

    A proto-instance is a tuple which contains the component it represents, its init argument (which can be anything), and a list of destinations, arranged by out port.

    Link to this function sources_match?(workflow, kw_list) View Source

    Check if a keyword lists matches with the sources of a workflow.

    This means that the keys of the keyword list and source names should be identical. Ordering is not taken into account.

    Examples 

    iex> sources_match?(example_workflow(), s1: 3, s2: 4)
true
iex> sources_match?(example_workflow(), s1: 3, s1: 4)
false
iex> sources_match?(example_workflow(), s1: 3)
false
